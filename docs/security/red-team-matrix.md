# Red-Team Matrix

This matrix maps in-scope adversaries to concrete controls, tests, evidence, and non-goals. The wording is intentionally bounded: the controls reduce risk and make drift visible, but they do not eliminate all failure modes.

| Adversary | Controls | Tests / Evidence | Non-Goals |
| --- | --- | --- | --- |
| malformed artifact producer | Reject malformed JSON and non-regular files early; cap manifest and payload sizes; require schema and version checks before deeper verification. | Negative-path loader tests; oversized-manifest checks; fuzz smoke for manifest parsers; oracle checks for bound-state replay. | Does not prevent a malicious producer from generating arbitrary junk off-chain. |
| recomputed bad-artifact producer | Bind artifacts to semantic scope, statement version, backend version, and registry commitments; reject mismatched nested proofs and replayed payloads. | Tamper-path tests for wrong references, broken links, and wrong version fields; oracle/differential verification on the same manifest state. | Does not prove the recomputed artifact was not produced elsewhere with the same labels. |
| model-substitution | Record model identity and scope in the manifest; keep model-specific claims separate from generic proof claims. | Claim-evidence ledger entries; documentation scope checks; paper-side provenance review. | Does not authenticate the original model provider. |
| tokenizer-substitution | Treat tokenizer identity as a first-class provenance field when tokenizer-sensitive artifacts are involved. | Documentation and manifest review for tokenizer-bound flows; regression checks for tokenizer-dependent layouts when present. | Does not detect a substitution that leaves all observable outputs unchanged. |
| quantization-substitution | Bind quantization mode and precision assumptions into the artifact scope when quantized flows are claimed. | Config and manifest checks; documentation review for quantized vs non-quantized claims. | Does not prove the quantization scheme is optimal or unique. |
| cached-response | Require input-linked commitments and step ordering so replayed outputs are not silently accepted as fresh. | Public-input ordering checks; replay-link tests; state-progress oracle checks. | Does not defend against an attacker who can replay a legitimately old response in a different context that is not bound. |
| public-input-ordering | Bind ordering into commitments and manifest validation so equivalent sets of inputs are not accepted in a different sequence. | Tests for broken history links, tampered rollup starts, reordered step references, and layout/total mismatches. | Does not claim that all permutations are inherently invalid; only the documented canonical forms are accepted. |
| backend-confusion | Keep backend family, proof backend version, and statement scope explicit; reject cross-family reuse. | Tests that reject vanilla vs `stwo` mismatches, unsupported backend versions, and backend-schema/version compatibility drift; this is narrower than the separate `schema-drift` adversary below. | Does not make different backends interchangeable. |
| schema-drift | Freeze manifest schema and loader expectations with versioned fields and bounded loaders. | Serialization round-trip tests; save/load budget tests; oracle checks against the same schema version. | Does not guarantee future schema extensions will be backward compatible without new work. |
| provenance-forgery | Freeze evidence snapshots, record hashes, and keep provenance metadata separate from commentary. | Evidence bundle hashes; frozen artifact bundles; repository review of quoted provenance. | Does not authenticate third-party pages beyond the archived snapshot. |
| paper-overclaim | Keep the paper and engineering docs scoped to what the repo actually proves. | Scope-limited paper language; claim-evidence ledger; explicit non-goals in docs. | Does not prevent readers from misreading a claim if they ignore the caveats. |

## Shared Controls

The following controls cut across multiple adversaries:

- versioned manifest fields
- semantic-scope binding
- resource caps for manifests and payloads
- oracle/differential tests for replay logic
- fuzz smoke for malformed inputs
- narrow formal-contract checks for the trusted core
- frozen evidence bundles for publication support

## Evidence Boundaries

Evidence in this repository is strongest when it is:

- generated by the repo itself
- backed by deterministic hashes
- tied to a specific backend version and manifest version
- paired with a negative-path test that shows what is rejected

Evidence is weaker when it depends on:

- mutable third-party pages
- broad narrative claims without a corresponding manifest or test
- assumptions about external systems that are outside this repository
