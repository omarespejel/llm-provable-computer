# zkAI d64 Stwo vector-row surface probe - 2026-05-01

## Question

For the canonical `d=64` RMSNorm-SwiGLU-residual statement, is the next Stwo
native proof blocked by arithmetic size, or by binding the private witness rows
to the committed model/table statement?

## Result

GO for the arithmetic row surface and for a proof-facing public-instance
contract. NO-GO-YET for an exact native Stwo proof.

The arithmetic surface is bounded and fits comfortably in the M31 lane for the
checked fixture:

- width: `64`
- feed-forward dimension: `256`
- projection multiplication rows: `49,152`
- trace rows excluding the static activation table: `49,920`
- bounded activation table rows: `2,049`
- max absolute checked intermediate: `849,454`
- signed M31 absolute limit used by the probe: `1,073,741,823`

This means the d64 target is not obviously blocked by arithmetic magnitude or
row count. The latest probe also fixes the next proof-facing public-instance
contract: the contract binds `proof_native_parameter_commitment`,
model-config, normalization, activation lookup, input, output,
verifier-domain, backend-version, public-instance commitment, and statement
commitment. Its mutation suite rejects `14 / 14` checked relabeling attempts.

That is useful, but it is not an exact proof. The remaining proof blocker is
relation-level commitment consistency: a native proof must show that the weight
rows and activation-table rows used inside the AIR match the committed
proof-native parameter identity, not merely carry that identity as public
instance data.

## Why This Matters

This rules out the wrong next optimization target. If we chase smaller
arithmetic rows before solving commitment consistency, we can produce a proof
that looks transformer-shaped but does not prove the same statement the receipt
claims.

The commitment-consistency method is now chosen: keep publication hashes for
audit/export identity and add a proof-native parameter commitment for native
proving. The next useful implementation PR should consume that proof-native
parameter commitment inside the AIR/export relation and then add proof-level
relabeling tests.

Only after that should the repo claim a proved `d=64` transformer block.

## Non-Claims

This is not:

- a Stwo proof,
- a verifier-time benchmark,
- proof that the d64 statement is accepted,
- relation-level consumption of `proof_native_parameter_commitment`,
- backend independence evidence,
- full transformer inference,
- a claim that commitment consistency is solved.
