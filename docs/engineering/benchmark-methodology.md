# Benchmark Methodology

This note defines the benchmark evidence protocol for issue `#153`.

## Scope

The benchmark harness is evidence-oriented, not a performance leaderboard.
Paper-facing claims should never cite a single run in isolation.

The preferred evidence unit is a JSON bundle generated by
`benchmarks/run_benchmarks.py` together with its sidecar logs and the exact
case manifest used for the run.

The bundle must validate with `benchmarks/validate_benchmark_result.py` before
any number from it is cited. Validation checks the result schema identity,
stable command hashes, stdout/stderr log hashes, and declared artifact hashes.

## Required metadata

Every benchmark record should capture:

- git SHA
- toolchain
- hardware
- OS
- command
- stable `command_sha256` over the command binding
- input hashes for executed `run` records
- declared output-artifact hashes for executed `run` records
- run mode (`dry-run` or `run`)
- dry-run records are execution plans; the harness calls
  `describe_inputs(case.inputs, hash_files=not dry_run)`, so dry-run records
  keep declared input paths but do not read inputs or produce input hashes,
  stdout hashes, or stderr hashes
- repetition count
- stdout/stderr hashes captured by the harness for executed `run` records
- `log_sha256` binding the stdout/stderr hashes and run status
- output artifact hashes for every declared case output

If a metric is unavailable, record `null` and explain why in the case notes or
bundle README instead of inventing a placeholder value.

## Timing protocol

- Do not write single-run timing numbers into the paper unless the command is
  clearly one-off and non-performance-sensitive.
- When a command is cheap and stable enough to repeat, record `p50` and `p95`
  in addition to the raw run list.
- Use repeated runs for:
  - schema validation time
  - verifier time
  - known-bad corpus runner time
  - mutation score collection
  - any other deterministic, short-lived evidence command
- Use one-shot runs only for expensive artifact generation paths that would be
  misleading to repeat.

## Metric definitions

The harness and surrounding evidence bundles should be able to represent the
following metrics.

| Metric | Definition | Notes |
| --- | --- | --- |
| Artifact generation time | Wall-clock time to construct a benchmark artifact or evidence bundle. | Record the command, command hash if used, and any input hashes. |
| Verifier time | Wall-clock time spent verifying a generated artifact. | Prefer repeated runs when the verifier is deterministic. |
| Proof/receipt size | Serialized size in bytes of the generated proof or receipt. | Record the file hash and size together. |
| Trace rows | Number of rows in the proof trace or witness trace. | Record the source artifact and any extraction rule. |
| Symbolic constraints | Count of symbolic constraints or algebraic relations used by the backend. | This is a structural metric, not a runtime substitute. |
| Peak RSS/VRAM | Peak resident set size or GPU memory usage during the run. | Record the unit used by the collector; `null` if unavailable. |
| Public input count | Number of public inputs exposed by the artifact. | Include the counting convention in the bundle note. |
| Commitment count | Number of commitments emitted by the artifact or verifier. | Keep the counting convention stable across runs. |
| Lookup table size | Total size or row count of lookup tables. | Note whether the count is per-table or aggregated. |
| Hash count | Number of hash invocations or hash objects, depending on the evidence source. | Do not mix backend hash calls with artifact hashes without labeling them. |
| Schema validation time | Time spent validating JSON/schema/manifest structure. | Useful for rejecting malformed or oversized inputs. |
| Known-bad corpus runner time | Time spent running negative corpus samples through parsers or verifiers. | This should fail closed on malformed cases. |
| Fuzz corpus size | Number of retained fuzz inputs in the corpus. | Report before and after corpus minimization when applicable. |
| Mutation score | Mutants killed divided by total mutants attempted. | Record the toolchain and mutant selection scope. |

## Execution guidance

- Keep the default harness mode in dry-run so the repository stays safe.
- Only run `--run` for commands that are explicitly intended to execute.
- Do not route heavyweight prover runs through the harness unless the command
  has already been validated locally.
- For commands that produce reusable evidence, store the JSON output under
  `benchmarks/results/` and keep the companion logs next to it.

## Recommended output contents

The JSON bundle should include:

- case manifest hash
- harness hash
- benchmark result schema hash
- git metadata
- toolchain metadata
- host metadata
- per-case command lists
- per-case command hashes
- per-run durations
- per-run stdout/stderr hashes
- per-run log-binding hashes
- per-run output paths
- declared output artifact paths, sizes, and hashes
- optional rusage fields when the platform provides them

## How to interpret the data

- Use `p50` as the representative repeated-run timing.
- Use `p95` to show variance and tail behavior.
- Use the raw run list to audit outliers.
- Use hashes and metadata to keep the run reproducible and auditable.

## Non-goals

- This benchmark slice is not a substitute for proof soundness tests.
- It does not replace fuzzing, mutation testing, Miri, sanitizers, or formal
  checks.
- It is not intended to run heavyweight prover workloads by default.
